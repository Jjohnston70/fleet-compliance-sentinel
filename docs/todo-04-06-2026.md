## TASK 1: C-2 -- Server-Side Assessment Score Validation

**Severity:** CRITICAL
**Agent:** `backend-developer` (`.claude/agents/backend-developer.md`)
**Audit Finding:** `docs/integration/TRAINING_GATEWAY_AUDIT_2026-04.md` section C-2
**Blocks:** First paying client -- regulatory compliance liability

### Context

The assessment submit endpoint at `src/app/api/v1/training/[moduleCode]/assessment/submit/route.ts` currently trusts client-provided `score`, `total`, `percentage`, and `passed` values. The client computes the final score in `src/components/training/TrainingAssessment.tsx` (lines 137-165) and posts it directly. The server validates types but does NOT recompute correctness. This means any authenticated user can submit a fabricated passing payload and generate a fraudulent DOT/FMCSA compliance completion record.

### Constraints

- MUST recompute score server-side from the authoritative answer key stored in `knowledge/training-content/assessments/` JSON files.
- MUST ignore client-provided `score`, `percentage`, and `passed` for all persistence decisions.
- Client-provided `answers` array is the ONLY trusted input (the user's selected answer indices/values).
- Answer key files are in `knowledge/training-content/assessments/{moduleCode}.json` -- each has a `questions` array with `correct_answer` or `correct_index` fields.
- The server must load the answer key, compare each submitted answer against the correct answer, compute score/total/percentage/passed, and use those server-computed values for all DB writes.
- Passing threshold is defined per training plan (default 80%) -- read from `training_plans` table `passing_score` column.
- Do NOT change the client-side TrainingAssessment.tsx display logic (it can still compute locally for instant feedback) -- just override on the server.
- Every query must include `org_id`. No exceptions.
- No PII in logs. Log moduleCode + employeeId (IDs only).
- Emit `audit.training.assessment.server_validated` event with `{ orgId, moduleCode, serverScore, serverPassed, clientClaimedPassed }` for drift detection.

### Deliverables

1. Modified `src/app/api/v1/training/[moduleCode]/assessment/submit/route.ts` with server-side scoring.
2. New utility: `src/lib/training-scoring.ts` -- `computeServerScore(moduleCode: string, submittedAnswers: Answer[]): { score: number, total: number, percentage: number, passed: boolean }`.
3. Answer key loading function that reads from the assessment JSON files.
4. Audit event emission on every scored submission.
5. Test file: `src/lib/training-scoring.test.ts` covering correct scoring, edge cases (empty answers, wrong moduleCode, threshold boundary).

### Output

- Modified files: `src/app/api/v1/training/[moduleCode]/assessment/submit/route.ts`
- New files: `src/lib/training-scoring.ts`, `src/lib/training-scoring.test.ts`
- Test command: `tsx --test src/lib/training-scoring.test.ts`
- Verification: Submit a correct assessment payload, confirm server score matches; submit an incorrect one, confirm server rejects even if client claims `passed: true`.

## TASK 2: C-1 -- Training Schema Deployment Safety

**Severity:** CRITICAL
**Agent:** `database-designer` (`.claude/agents/database-designer.md`)
**Audit Finding:** `docs/integration/TRAINING_GATEWAY_AUDIT_2026-04.md` section C-1
**Blocks:** New org onboarding -- training features return 500s

### Context

Training tables (`training_plans`, `training_assignments`, `training_progress`) are created only in `migrations/012_training_tables.sql`. The app routes query these tables directly with no schema guard. `scripts/init-db.mjs` provisions only the `chief_records` table (lines 28-41) and does NOT run the full migration pipeline. New orgs or fresh database deployments will hit live 500 errors on all training API routes.

### Constraints

- All 19 migration files are in `migrations/` (001 through 019).
- Migrations use `CREATE TABLE IF NOT EXISTS` in some cases but not consistently.
- `scripts/init-db.mjs` must be updated to run ALL migrations in sequence, not just `chief_records`.
- Every migration must be idempotent -- use `IF NOT EXISTS` for CREATE TABLE, `IF EXISTS` for ALTER/DROP.
- Do NOT change table schemas or column definitions -- only make the creation idempotent.
- Do NOT drop or recreate existing tables with data.
- Add a preflight health check function that asserts required training tables exist before the training module serves requests.
- The health check should be callable from `src/app/api/v1/training/health/route.ts`.
- Database is Neon PostgreSQL. Connection via `DATABASE_URL` env var.
- Every query scoped to `org_id`. Migrations are schema-level (no org_id needed on DDL).

### Deliverables

1. Updated `scripts/init-db.mjs` -- runs all 19 migration files in order, idempotently.
2. Audit of all 19 migration files for idempotency -- add `IF NOT EXISTS` / `IF EXISTS` guards where missing.
3. New utility: `src/lib/training-preflight.ts` -- `assertTrainingTablesExist(): Promise<{ ready: boolean, missing: string[] }>`.
4. Updated `src/app/api/v1/training/health/route.ts` -- returns 503 with missing table list if preflight fails.
5. Test: `src/lib/training-preflight.test.ts` validating the check logic.

### Output

- Modified files: `scripts/init-db.mjs`, `migrations/*.sql` (only adding IF NOT EXISTS guards), `src/app/api/v1/training/health/route.ts`
- New files: `src/lib/training-preflight.ts`, `src/lib/training-preflight.test.ts`
- Verification: Run `node scripts/init-db.mjs` twice consecutively -- second run must succeed with no errors. Call `/api/v1/training/health` -- must return 200 with `{ ready: true }`.

### Deliverables

2. Guard queries: `SELECT id, status FROM training_assignments WHERE org_id = $1 AND employee_id = $2 AND module_code = $3 AND status IN ('active', 'in_progress')` and `SELECT deck_completed_at FROM training_progress WHERE assignment_id = $1 AND deck_completed_at IS NOT NULL`.

4. Audit event for blocked attempts.

### Output

**Audit Finding:** `docs/integration/TRAINING_GATEWAY_AUDIT_2026-04.md` section C-4
**Depends on:** Task 1, Task 3

### Context

The assessment submit endpoint catches all persistence errors (lines 305-308) and still returns `201` with the client-provided result. This means the UI shows "passed" but the database may have partial/missing records (no compliance record, no certificate, no progress update).

### Constraints

- Wrap the entire persistence block (progress update, compliance record write, certificate generation, notification) in a PostgreSQL transaction.
- If ANY write in the transaction fails, ROLLBACK and return `500` with `{ error: "Assessment result could not be persisted. Please retry." }`.
- Do NOT return 2xx when persistence fails.
- Log the error to Sentry (no PII -- log orgId, moduleCode, error message only).
- The existing catch block that swallows errors must be replaced with fail-closed behavior.
- Use `BEGIN` / `COMMIT` / `ROLLBACK` with the Neon SQL driver.

### Deliverables

1. Modified `src/app/api/v1/training/[moduleCode]/assessment/submit/route.ts` -- transaction wrapper around all persistence operations.
2. Proper error propagation and 500 response on failure.
3. Sentry error capture with safe metadata.

### Output

- Modified files: `src/app/api/v1/training/[moduleCode]/assessment/submit/route.ts`
- Verification: Simulate a DB write failure (e.g., temporarily invalid column) -- endpoint must return 500, not 201. Normal flow must still return 201 with correct data.

## TASK 9: H-4 -- Durable Certificate Storage

**Audit Finding:** `docs/integration/TRAINING_GATEWAY_AUDIT_2026-04.md` section H-4 and `.claude/ARCHITECTURE.md` known gaps

### Context

Training certificates are stored on Vercel's ephemeral filesystem (`storage/{org_id}/training-certs/`). Certificates are lost on every redeploy. The certificate generation uses `pdf-lib` in `src/lib/training-certificate.ts` and writes to local disk.

### Constraints

- Migrate certificate storage to Neon PostgreSQL as base64-encoded blobs in a `training_certificates` table.
- Table schema: `id`, `org_id`, `employee_id`, `module_code`, `certificate_pdf` (bytea or text/base64), `issued_at`, `expires_at`, `created_at`.
- Retain the existing `pdf-lib` generation logic -- just change the storage target.
- The certificates API route (`src/app/api/v1/training/certificates/route.ts`) must read from the database instead of filesystem.
- Migration file: `migrations/020_training_certificate_storage.sql` -- `CREATE TABLE IF NOT EXISTS training_certificates (...)`.
- Every query scoped to `org_id`.
- Add `ON CONFLICT (org_id, employee_id, module_code) DO UPDATE` for re-issuance.
- Consider storage size: typical training cert PDF is 50-100KB, base64 is ~33% larger. Neon handles this fine for small fleet operators (5-20 employees).

### Deliverables

1. New migration: `migrations/020_training_certificate_storage.sql`.
2. Modified `src/lib/training-certificate.ts` -- write PDF bytes to database instead of filesystem.
3. Modified `src/app/api/v1/training/certificates/route.ts` -- read from database, serve as PDF response.
4. Modified assessment submit route -- update certificate storage calls.
5. Cleanup: remove filesystem write logic (or keep as fallback behind env flag `TRAINING_CERT_STORAGE_BACKEND=database|filesystem`).

### Output

- Modified files: `src/lib/training-certificate.ts`, `src/app/api/v1/training/certificates/route.ts`, assessment submit route
